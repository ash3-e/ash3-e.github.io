What happens when you type a prompt
01

CLAUDE.md loads

Claude reads your project rules, file map, agent role, and vexp instructions. Happens silently every session.

02

vexp run_pipeline fires

Claude calls vexp automatically per CLAUDE.md rules. Returns relevant code nodes, skeletons, and past observations.

03

Claude reads exact files

Only the files identified by vexp get opened. Not the whole repo. Not the 339K context dump.

04

Edits + observation saved

Claude writes to claude-code-* files only, then persists what it learned for next session.

What the script layers on top of vanilla Claude Code defaults

Plain Claude Code is capable, but on this repo it starts blind: no MCP, no file exclusions, no agent boundaries, no project memory. The setup script is the deliberate layer that fixes all of that in one run.

Vanilla Claude Code
What you get
Reads the repo with grep/cat/find. Loads CLAUDE.md if present (yours was empty). Discovers the project from scratch every session.
What is missing
No MCP tools, no file exclusions, no multi-agent awareness, no slash commands, no protection against 800K+ of context bombs sitting in root.
After setup script
What it adds
Auto-detected vexp MCP wiring, populated CLAUDE.md, real .claudeignore, project-local .mcp.json, and /svg /doc /review slash commands.
Why it matters
Claude enters each session with project memory loaded, junk files blocked, agent boundaries enforced, and a retrieval-first workflow baked in. You just prompt.
Quality boundary
Keep quality
vexp handles discovery. Claude still reads exact source files before editing. Compression lives in the discovery phase, not in the final source truth.
Lose quality
Treating vexp capsules as the source files themselves, or over-optimizing by skipping direct inspection before edits.

Guardrails worth keeping visible

These are the rules most likely to prevent subtle failures: cross-agent collisions, stale MCP paths, accidental context explosions, and overconfident edits from incomplete reads.

Do this

  1. Use /mcp to verify vexp is connected on first session after setup. If dismissed or failed, fix before prompting.
  2. Re-run setup-claude.ps1 when the vexp extension version changes so the MCP path stays current.
  3. Use /clear between unrelated tasks. Context from task 1 is dead weight in task 2.
  4. Use /compact at ~50% context usage within a long task. Do not wait until you hit the wall.
  5. Save decisions via vexp observations so the next session inherits them automatically.

Avoid this

  • Do not assume vexp capsules replace reading source. Use capsules for discovery, read files for edits.
  • Do not let Claude edit shared or other-agent files. CLAUDE.md prevents this, but verify on first run.
  • Do not keep dead MCP paths after updating the VS Code extension — re-run the setup script.
  • Do not mix unrelated tasks in one prompt. One job, one prompt, one /clear between.
  • Do not skip /clear and let context rot across task boundaries.

BCODe.meta warning

Treat BCODe.meta as an advanced concept. Do not route beginners into it by default. Use it when the task is explicitly about a higher-level design decision, shared structured pattern, or meta-layer behavior — not as the first stop for ordinary syntax or portal onboarding.
